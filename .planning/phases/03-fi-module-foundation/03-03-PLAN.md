# Financial Accounting — SPRO/IMG Configuration

> ECC 6.0 reference. All paths use the "Financial Accounting (New)" node (introduced with New GL in ECC 6.0). For Classic GL systems (pre-New GL migration), the root node is "Financial Accounting" without "(New)".

> **How to read this file:** Each config step shows the full IMG path string first, then the T-code shortcut in parentheses. Use the IMG path when navigating the tree manually; use the T-code to jump directly to the screen. Where no T-code shortcut exists, navigate via the IMG tree.

## GL Configuration

### Step 1: Define Chart of Accounts

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Master Data ▸ G/L Accounts ▸ Preparations ▸ Edit Chart of Accounts List (OB13)

**What to do:**
1. Create a new chart of accounts entry or copy from a reference chart (e.g., INT = International)
2. Set the Chart of Accounts key (4 characters, e.g., CAUS for USA operations)
3. Enter description and maintenance language
4. Set the account length (typically 6 or 10 digits)
5. Optionally assign a group chart of accounts for consolidation reporting

**Key fields:** KTOPL (chart of accounts key), XBILK indicator default, account length. Do not change account length after accounts are created.

### Step 2: Assign Chart of Accounts to Company Code

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Master Data ▸ G/L Accounts ▸ Preparations ▸ Assign Company Code to Chart of Accounts (OB62)

**What to do:**
1. Enter the company code
2. Enter the chart of accounts key created in Step 1
3. Optionally assign a country chart of accounts for local statutory reporting alongside the operational chart
4. Save

**Note:** One company code can have one operational chart of accounts plus one country-specific chart. Changing this assignment after accounts are created requires a complex migration.

### Step 3: Maintain Fiscal Year Variant

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Fiscal Year and Posting Periods ▸ Fiscal Year ▸ Maintain Fiscal Year Variant (Maintain Shortened Fiscal Year) (OB29)

**What to do:**
1. Create a fiscal year variant key (e.g., K4 = calendar year, V3 = April-March)
2. Define whether periods are calendar months or custom date ranges
3. For non-calendar fiscal years, define the period start/end dates
4. Define the number of special periods (usually 4, used for year-end adjustments)

**Note:** SAP delivers standard variants (K4 = calendar year, V3 = fiscal year April-March). Copy and modify rather than creating from scratch.

### Step 4: Assign Fiscal Year Variant to Company Code

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Fiscal Year and Posting Periods ▸ Fiscal Year ▸ Assign Company Code to a Fiscal Year Variant (OB37)

**What to do:** Select the company code and assign the fiscal year variant created in Step 3. Save.

### Step 5: Define Document Types

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Document ▸ Document Types ▸ Define Document Types for Entry View (OBA7)

**What to do:**
1. Review delivered document types (SA = GL posting, KR = vendor invoice, KZ = vendor payment, DR = customer invoice, DZ = customer payment, AA = asset posting)
2. For each document type, set: number range assignment, account types allowed, reversal document type, net posting (optional)
3. Create custom document types for business-specific needs using the same structure

**Key fields:** BLART (document type key), NRART (number range), allowed account types (XBLNR, BSCHL), reversal document type.

### Step 6: Define Posting Keys

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Document ▸ Line Item ▸ Controls ▸ Define Posting Keys (OB41)

**What to do:** Review and customize field status for each posting key. Common posting keys:
- 40 = GL debit; 50 = GL credit
- 01 = customer invoice (debit); 11 = customer credit memo (credit)
- 31 = vendor invoice (credit); 21 = vendor credit memo (debit)
- 70 = asset debit; 75 = asset credit

**Note:** SAP delivers all standard posting keys. Custom posting keys are rarely needed; modify field status (required/optional/hidden) for existing keys as needed.

### Step 7: Define Field Status Variants and Groups

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Fields ▸ Define Field Status Variants (OBC4)

**What to do:**
1. Create or copy a field status variant (e.g., copy 0001 = SAP standard)
2. Within the variant, define field status groups (e.g., G001 = general GL accounts, G004 = bank accounts)
3. For each group, configure fields as Required (R), Optional (O), or Hidden (H) across field categories: general data, additional account assignments (cost center, profit center), payment transactions

**Key link:** Field status groups (FSTAG in SKB1) are assigned per GL account. The group controls what fields appear when posting to that account.

### Step 8: Define GL Account Groups

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Master Data ▸ G/L Accounts ▸ Preparations ▸ Define Account Group (OBD4)

**What to do:**
1. Define account groups (e.g., 0001 = assets, 0010 = liabilities, 1000 = revenue)
2. For each group, assign the number range and field status group
3. Account group controls which fields are required/optional when creating GL accounts in FS00

### Step 9: Open/Close Posting Periods

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Fiscal Year and Posting Periods ▸ Posting Periods ▸ Open and Close Posting Periods (OB52)

**What to do:**
1. Enter the posting period variant
2. For each account type (A=assets, D=customers, K=vendors, S=GL, M=materials, += all types), set the From Period and To Period that are open
3. Save — effective immediately; no transport required for period opens in production

**Gotcha:** Period opening by account type allows selective opening (e.g., keep period open for GL but close for vendors). The "+" type row acts as a default; specific account type rows override it.

## AP/AR Configuration

### Step 1: Define Payment Terms

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Business Transactions ▸ Outgoing Invoices/Credit Memos ▸ Define Terms of Payment (OBB8)

**What to do:**
1. Create payment term key (e.g., 0001, Z030)
2. Set payment term description
3. Define baseline date calculation (document date, posting date, or fixed day of month)
4. Set up to 3 installments with days and discount percentages (e.g., 14 days 2% discount, 30 days net)
5. Assign default for accounts payable and/or accounts receivable

**Note:** Payment terms defined here are assigned in vendor master (LFB1.ZTERM) and customer master (KNB1.ZTERM). They drive due date calculation and cash discount posting in F110 and F-28.

### Step 2: Define Tolerance Groups for Employees

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Business Transactions ▸ Incoming Invoices/Credit Memos ▸ Define Tolerance Groups for Employees (OBA3 for G/L and vendor; OBA4 for customers)

**What to do:**
1. Define tolerance group key (blank = default group, applies to all users without explicit assignment)
2. Set upper limits for posting amounts per document and per line item
3. Set cash discount limits (amount and percentage the user is allowed to grant or accept)
4. Assign users to tolerance groups in OB57

**Key fields:** Upper limit for posting, permitted payment difference (amount/percentage), permitted cash discount amount. The blank tolerance group applies as the system default.

### Step 3: Define Dunning Procedures

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Business Transactions ▸ Dunning ▸ Dunning Procedure ▸ Define Dunning Procedures (FBMP)

**What to do:**
1. Create dunning procedure (e.g., 0001 = standard dunning)
2. Define number of dunning levels (e.g., 4 levels: reminder, first notice, second notice, legal action)
3. For each level, set: days overdue threshold, dunning text (form), minimum amounts, interest calculation
4. Set dunning interval (days between successive dunning runs)
5. Assign dunning procedure to customer master (KNB1)

**Note:** F150 executes the dunning program against this configuration. Dunning can also apply to vendors (for credit memos not yet applied).

### Step 4: Configure Automatic Payment Program (FBZP)

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Business Transactions ▸ Outgoing Payments ▸ Automatic Outgoing Payments ▸ Payment Method/Bank Selection for Payment Program ▸ Set Up All Company Codes for Payment Transactions (FBZP)

**What to do — FBZP is the central configuration hub for F110. It has five sub-areas:**

**1. All Company Codes** (FBZP → All Company Codes):
- Assign paying company code and sending company code
- Set outgoing payment tolerance, minimum payment amount
- Define the GL account for bills of exchange payable (if applicable)

**2. Paying Company Codes** (FBZP → Paying Company Codes):
- For the paying company code: assign forms, set minimum payment amounts per currency
- This step unlocks cross-company-code payment (one CC pays on behalf of another)

**3. Payment Methods in Country** (FBZP → Payment Methods in Country):
- Define payment methods per country (C = check, T = bank transfer, E = EFT)
- Specify the DME program for each method (bank file format)
- Set whether a print program is needed (check printing)

**4. Payment Methods in Company Code** (FBZP → Payment Methods in Company Code):
- Per company code + payment method: minimum and maximum amounts, bank optimization, document type
- Assign the form for check printing or bank file for electronic payments

**5. Bank Determination** (FBZP → Bank Determination):
- Rank available house bank accounts by currency and payment method
- Set available amounts for each bank account
- This controls which bank account F110 uses for each payment

**CRITICAL distinction:** FBZP is CONFIGURATION. F110 (transaction) is EXECUTION. You must complete FBZP configuration before F110 can run a payment proposal.

### Step 5: Define Vendor Account Groups

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Vendor Accounts ▸ Master Data ▸ Preparations for Creating Vendor Master Data ▸ Define Account Groups with Screen Layout (Vendors) (OBD3)

**What to do:**
1. Create or review vendor account groups (KRED = standard, LIEF = supplier/sub-contractor)
2. For each group, define the number range for vendor account numbers
3. Set field status per segment (general data, company code data, purchasing data): required/optional/hidden

### Step 6: Define Customer Account Groups

**IMG Path:** Financial Accounting (New) ▸ Accounts Receivable and Accounts Payable ▸ Customer Accounts ▸ Master Data ▸ Preparations for Creating Customer Master Data ▸ Define Account Groups with Screen Layout (Customers) (OBD2)

**What to do:** Same structure as OBD3 but for customer account groups (KUNA = standard customer, DEBTR = debtor). Field status here controls what fields are required when creating customer masters in XD01/FD01.

## Asset Accounting Configuration

> **CRITICAL — Configuration order:** Asset Accounting configuration steps must be executed in sequence. Doing them out of order causes system errors that are difficult to unwind. The mandatory sequence is:
> **EC08 → OAOB → OAOA → AO90 → AFAMA**

### Step 1: Copy Reference Chart of Depreciation (EC08)

**IMG Path:** Financial Accounting (New) ▸ Asset Accounting ▸ Organizational Structures ▸ Copy Reference Chart of Depreciation/Depreciation Areas (EC08)

**What to do:**
1. Copy a SAP reference chart of depreciation (e.g., 1US = US GAAP, 1DE = German HGB) to create your chart
2. The copy brings all depreciation areas predefined (01 = book, 10 = tax, 15 = IFRS, 20 = cost accounting)
3. After copying, review and activate/deactivate depreciation areas not needed for your country

**Note:** Never configure Asset Accounting without first completing this step. The chart of depreciation defines the universe of depreciation areas available to all asset classes and assets.

### Step 2: Assign Chart of Depreciation to Company Code (OAOB)

**IMG Path:** Financial Accounting (New) ▸ Asset Accounting ▸ Organizational Structures ▸ Assign Chart of Depreciation to Company Code (OAOB)

**What to do:**
1. Select the company code
2. Assign the chart of depreciation created in Step 1
3. Set the asset fiscal year (usually same as FI fiscal year variant)

**Note:** One company code = one chart of depreciation. This assignment cannot be changed after assets are created.

### Step 3: Define Asset Classes (OAOA)

**IMG Path:** Financial Accounting (New) ▸ Asset Accounting ▸ Organizational Structures ▸ Asset Classes ▸ Define Asset Classes (OAOA)

**What to do:**
1. Create asset classes (e.g., 1000 = Buildings, 2000 = Vehicles, 3000 = Machinery and Equipment, 4000 = Low-Value Assets)
2. For each class, assign: number range, screen layout (controls required/optional fields in AS01), account determination key
3. Set the default depreciation terms (depreciation key, useful life) that new assets inherit

**Note:** Asset class drives the default G/L accounts (via account determination in AO90) and the default depreciation key. Users can override depreciation key per asset but not the GL accounts.

### Step 4: Specify G/L Account Determination (AO90)

**IMG Path:** Financial Accounting (New) ▸ Asset Accounting ▸ Integration with the General Ledger ▸ Assign G/L Accounts (AO90)

**What to do:**
1. For each combination of chart of depreciation + account determination key + depreciation area:
2. Assign the GL accounts for: asset balance sheet account, accumulated depreciation account, depreciation expense account, asset retirement gain/loss accounts, clearing accounts for acquisitions
3. Save for each depreciation area separately

**Note:** AO90 requires SKB1 to exist for all assigned GL accounts in the company code. If GL accounts don't exist yet, create them in FS00 first. This is the integration point between AA and GL — every asset posting (acquisition, depreciation, retirement) hits the GL accounts defined here.

### Step 5: Maintain Depreciation Keys (AFAMA)

**IMG Path:** Financial Accounting (New) ▸ Asset Accounting ▸ Depreciation ▸ Valuation Methods ▸ Depreciation Key ▸ Maintain Depreciation Key (AFAMA)

**What to do:**
1. Review delivered depreciation keys (LINR = straight-line to zero, DBNL = declining balance, GD01 = straight-line from net book value)
2. For each depreciation key, define the calculation method: base method, declining balance method, period control
3. Create custom depreciation keys only when delivered keys don't match local statutory requirements

**Note:** Depreciation keys are referenced in asset class defaults (OAOA) and in ANLB per asset. The AFAB depreciation run uses these keys to calculate periodic depreciation amounts.

## New GL Configuration

> **Assumption:** New GL is active (standard for ECC 6.0 implementations post-2008). If Classic GL is still active, the "Financial Accounting (New)" SPRO node may not show the New GL sub-nodes below.

### Step 1: Activate New General Ledger Accounting

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Ledger ▸ Activate New General Ledger Accounting

**What to do:** Set the activation indicator for the client. This is a one-time, irreversible activation. Once New GL is active, Classic GL tables (GLT0) stop being updated. Plan the activation carefully — migration tooling is available if moving from Classic GL.

**Warning:** New GL activation is client-wide. There is no per-company-code activation (only deactivation).

### Step 2: Define Ledgers for General Ledger Accounting

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Ledger ▸ Define Ledgers for General Ledger Accounting

**What to do:**
1. The leading ledger (0L) is automatically created. Assign the leading ledger to the appropriate accounting principle (IFRS, US GAAP, local GAAP)
2. Ledger 0L stores all postings to FAGLFLEXT and FAGLFLEXP
3. Only one leading ledger is allowed per client

### Step 3: Define and Activate Non-Leading Ledgers

**IMG Path:** Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Ledger ▸ Define and Activate Non-Leading Ledgers

**What to do:**
1. Create non-leading ledgers for parallel accounting (e.g., ledger "L1" for local GAAP, "T1" for tax reporting)
2. Assign each non-leading ledger to an accounting principle
3. Activate the non-leading ledger for the relevant company codes

**Use case:** Company reporting under both IFRS (leading ledger) and local GAAP (non-leading ledger) maintains separate depreciation values via separate ledger assignments per depreciation area in OAOB/ANLB.

### Step 4: Activate Document Splitting

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Business Transactions ▸ Document Splitting ▸ Activate Document Splitting

**What to do:**
1. Activate document splitting for the client (global activation)
2. Select the splitting method (standard method 0000000012 in most implementations)
3. Mark "Inheritance" if passive splitting (inheriting segment/profit center from leading item) is required

**CRITICAL:** Document splitting activation is **client-level**. If you activate it, it applies to all company codes. To exclude specific company codes, use the per-company-code deactivation step (Financial Accounting (New) ▸ Financial Accounting Global Settings (New) ▸ Ledgers ▸ Ledger ▸ Deactivate Document Splitting per Company Code).

### Step 5: Define Document Splitting Characteristics

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Business Transactions ▸ Document Splitting ▸ Define Document Splitting Characteristics for General Ledger Accounting

**What to do:**
1. Select the characteristics that must be split on every document line (commonly: Segment, Profit Center, Business Area)
2. For each characteristic, indicate whether zero balance is required (if yes, a zero-balance clearing account is needed for documents where the characteristic doesn't naturally balance)

**Required vs optional:**
- **Required:** Classify GL accounts (Step 6) and document types (Step 7) before document splitting will work correctly
- **Required for zero balance:** Define a zero-balance clearing account (GL account posted automatically when lines don't naturally sum to zero per characteristic)
- **Optional:** Mandatory field flag on characteristics; custom splitting methods

### Step 6: Classify G/L Accounts for Document Splitting

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Business Transactions ▸ Document Splitting ▸ Classify G/L Accounts for Document Splitting

**What to do:** Assign each GL account to an item category (e.g., 01000 = balance sheet accounts, 02000 = customer receivables, 03000 = vendor payables, 04000 = cash accounts). The item category determines how the account participates in document splitting rules.

### Step 7: Classify Document Types for Document Splitting

**IMG Path:** Financial Accounting (New) ▸ General Ledger Accounting (New) ▸ Business Transactions ▸ Document Splitting ▸ Classify Document Types for Document Splitting

**What to do:** Assign each document type to a business transaction and business transaction variant (e.g., document type SA → business transaction 0000 = unspecified posting, KR → 0300 = vendor invoice). This controls which splitting rule the system applies.
