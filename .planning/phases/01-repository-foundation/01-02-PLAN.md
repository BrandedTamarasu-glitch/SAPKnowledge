**CLI interface:**
```
Usage: python validate.py [--strict] [path...]
  --strict: Treat warnings as errors (exit 2)
  path...:  Files or directories to check (default: modules/, cross-module/, reference/)
Exit codes: 0 = pass, 1 = critical errors, 2 = warnings (with --strict)
```

**Check 1: Frontmatter presence and required fields (CRITICAL)**
- Every .md file in modules/, cross-module/, reference/ must have YAML frontmatter
- Required fields: module, content_type, ecc_version, ehp_range, confidence, last_verified
- Skip CLAUDE.md files in subdirectories (they have different required fields — still need module, content_type, ecc_version, but ehp_range/confidence/last_verified are also required per convention)

**Check 2: Frontmatter value validation (CRITICAL)**
- `module` must be one of: fi, mm, sd, co, cross-module, reference
- `content_type` must be one of: index, tcodes, config-spro, processes, master-data, integration, patterns, e2e-process, lookup-table
- `ecc_version` must be "6.0"
- `ehp_range` must match pattern like "0-8", "6-8", or single digit
- `confidence` must be one of: high, medium, low
- `last_verified` must be a valid ISO date (YYYY-MM-DD)

**Check 3: S/4HANA contamination detection (CRITICAL for unambiguous, WARNING for ambiguous)**
- Scan content body (not frontmatter, not fenced code blocks) for S/4-only terms
- CRITICAL (hard block) patterns: Universal Journal, ACDOCA, MATDOC, Simplified Data Model
- WARNING (flag for review) patterns: Business Partner (outside CRM context), CDS Views (as standard feature), Embedded Analytics, Fiori Launchpad, Manage [X] (Fiori app names), BP transaction (as vendor/customer maintenance)
- Use the regex patterns from research Section 5. Exclude content inside fenced code blocks (```...```) and inside blockquotes starting with "> **S/4HANA difference:**" (those are intentional disambiguation callouts)

**Check 4: Token budget (WARNING)**
- Estimate tokens as len(content) // 4
- Define budget by content_type:
  - index: hard cap 600
  - tcodes, config-spro: hard cap 5000
  - processes, master-data, patterns: hard cap 4000
  - integration: hard cap 3000
  - e2e-process: hard cap 5000
  - lookup-table: hard cap 5000
- Also check .claude/rules/ files: combined hard cap 1500 tokens for all files in that directory

**Check 5: Cross-reference integrity (WARNING)**
- Find all @path/to/file references in content
- Resolve relative paths from the containing file's directory
- Report any references that point to non-existent files
- This is warning-only because files may not exist yet during phased development

Create `.githooks/pre-commit` as a bash script:
```bash
#!/usr/bin/env bash
# Pre-commit hook for SAP Knowledge Base validation
# Runs validate.py on staged .md files

# Get list of staged .md files (added, copied, modified — not deleted)
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM -- '*.md')

if [ -z "$STAGED_FILES" ]; then
    exit 0  # No .md files staged, nothing to validate
fi

# Run validation on staged files only
python3 scripts/validate.py $STAGED_FILES
EXIT_CODE=$?

if [ $EXIT_CODE -ne 0 ]; then
    echo ""
    echo "Commit blocked: validation errors found. Fix the issues above and try again."
    echo "To bypass (not recommended): git commit --no-verify"
fi

exit $EXIT_CODE
```

Make it executable: `chmod +x .githooks/pre-commit`

Configure git to use the custom hooks directory:
```
git config core.hooksPath .githooks
```

This is a project-local git config, not global. It persists in .git/config for this repo only.
